Facettierte Suche:

Ergebnisse einer (Stichwort-)Suche, "Facetten" mit Zähler, verwendbar als Einschränkungen ("Filter")

"Facette" = Kategorie, "Facettenwerte" ("Constraints")

"Ausklappen von Facetten", Gestaltung der Facettenwerte nach Eigenschaft z.B. Zeitbereiche, Preisklassen, Taxonomien, Geographie

Visuelle Notationen: Breadcrumbs (Geschichte der der Reihe nach ausgewählten Filter), Clickboxen, Navbar mit Accordion

Filter = Zusammensetzung von Constraints

  • Einschränkung = AND (zwischen Facetten)
  • OR = z.B. Buch in Bibliothek X oder Bibliothek Y

"A query is typically an AND of ORs: values selected within a single facet are combined using a logical OR, whereas constraints associated with different facets are combined using a logical AND."

Ab wann ist eine Facette sinnvoll?

  • wenn ausreichend spezifische Facette (weiblich/männlich vs. Beruf einer Person)
    • abhängig von der Forschungsfrage?
  • vollständig befüllte Facetten
    • Datumsangaben (YYYY, YYYY-MM, YYYY-MM-DD)
    • "Undefined" als Wert ermöglichen
  • Zähler als wichtigen Gewinn bei facettierter Suche, da der Umfang eines Wertes angegeben wird

Projektspezifische facettierte Suche vs. GAMS übergreifende Suche

Bread Crumbs = Zweig eines Baums oder Reihenfolge der ausgewählten Filter

Dynamische Facetten nach Objektart (Bücher haben Seitenzahlen, Museumsobjekte haben Materialien, …)

Konzept:

  • Es gibt eine zentrale Config/Settings, wo die Endpoints definiert werden, die Facetten + die zugehörigen $Parameter für die SPARQLs, sowie die Interface Komponente, die im Search Form dafür angezeigt werden soll.

    Zur Zeit ist das sehr flach am Beginn des Hooks definiert, wird aber in eine eigenes File/eine eigene Komponente ausgelagert werden (eigtl. das nächste Todo) [https://zimlab.uni-graz.at/gams/frontend/react/apps/facetedsearch/-/blob/lindesc/sparql/src/hooks/useQueryObject.tsx](https://zimlab.uni-graz.at/gams/frontend/react/apps/facetedsearch/-/blob/lindesc/sparql/src/hooks/useQueryObject.tsx)

  • Es gibt unterschiedliche Komponenten für das Search Form, die dynamisch aufgrund des in der Config angegebenen Wertes "geladen" werden. Die Komponenten werden an zentraler Stelle gesammelt: https://zimlab.uni-graz.at/gams/frontend/react/apps/facetedsearch/-/tree/lindesc/sparql/src/components/common

    Die verwendeten Komponenten müssen auf einen Key gemappt werden, der dann in der Config angegeben werden kann. Am Beginn des folgenden Files (const ComponentsKeyMap): [https://zimlab.uni-graz.at/gams/frontend/react/apps/facetedsearch/-/blob/lindesc/sparql/src/components/App/Search/index.tsx](https://zimlab.uni-graz.at/gams/frontend/react/apps/facetedsearch/-/blob/lindesc/sparql/src/components/App/Search/index.tsx) Zur Zeit gibt es nur eine Komponente "Accordion" (eine Select Komponente wäre das nächste) Die Search Form Componenten bekommen folgende Properties: `toggleFacetField`, `facet`, `FacetQueryObject`, `facetFields`, und müssen im Endeffekt für jedes ausgewählte Feld die Funktion `toggleFacetField(facet, field)` auslösen

  • Es gibt zur Zeit 2 SPARQL Queries, die jedes Mal ausgelöst werden, wenn ein neues Feld in einer Facette ausgewählt wird:

  1. http://glossa.uni-graz.at/query:eltec.facet-fields/QUERY
  • Gibt alle vorhandenen Felder für eine Facette zurück + Count (basierend auf der aktuellen Auswahl im Search Form)
  1. http://glossa.uni-graz.at/query:eltec.db/QUERY

  • Gibt alle Resultate zurück (basierend auf der aktuellen Auswahl im Search Form)

    Die Bodies der beiden Queries müssen identisch sein. Aber, bei facet-fields ist es möglich, einzelne Facetten auf optional zu setzen, damit insgesamt auch Felder angezeigt werden, bei denen einzelne Triple nicht vorhanden sind. Das Binding der Values muss in diesem Fall nach dem optionalen Pattern erfolgen, Beim Result sollte das von der Logik her nicht vorkommen. Gezählt wird zur Zeit im Js, wäre aber sinnvoll dafür auch eine eigene Query zu machen - auch um Wartezeit zu verkürzen, wenn die eigentliche Anfrage etwas länger dauert. Die Implementierung von Slidern oder Textsuche könnte evtl. etwas komplexer werden. Das muss man sich aber anschauen.

  • Das Result wird zur Zeit noch hardgecoded ausgegeben, da könnte/sollte man auch irgendeine Art von Konfiguration andenken

    [https://zimlab.uni-graz.at/gams/frontend/react/apps/facetedsearch/-/blob/lindesc/sparql/src/components/App/Result/index.tsx](https://zimlab.uni-graz.at/gams/frontend/react/apps/facetedsearch/-/blob/lindesc/sparql/src/components/App/Result/index.tsx) Auch todo: Breadcrumbs und eigtl.
